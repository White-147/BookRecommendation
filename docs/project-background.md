# 项目背景与数据说明

本项目是本科毕业设计项目，主题为图书推荐系统。项目目标是围绕高校图书馆业务数据，完成用户登录注册、图书检索、借阅收藏、行为采集、离线/准实时推荐计算和推荐结果展示等功能。

从代码结构看，项目不是单一 Web 应用，而是由前端、后端、大数据计算和爬虫辅助工具组成的完整系统：

- `frontend`：Vue 2 前端，用于用户端和管理端页面展示。
- `backend`：Spring Boot 后端，用于业务接口、权限认证、MySQL 数据访问和用户行为日志写入 Kafka。
- `bigdata`：Spark/Hive/Kafka 计算模块，用于处理用户行为并生成推荐结果。
- `crawler`：Selenium 爬虫，用于补全图书封面等缺失信息。

## 原始数据来源

毕业设计时期使用的数据包含两类真实业务数据：

- 学校学生学籍相关数据：包含真实学号、姓名、学院班级等信息。
- 图书馆真实馆藏数据：包含真实图书编号、书名、作者、出版社、出版年份、馆藏状态等信息。

由于这些数据包含真实学生学号和学籍信息，不适合直接公开到 GitHub。即使数据库仍然存在，也应先完成脱敏或构造演示数据后再公开。

## 数据库缺失原因

当前仓库不包含原始 MySQL 数据库文件和完整 SQL 初始化数据。原因是项目完成至今已经约三年，中途因工作需要长期使用同一台电脑，在清理电脑数据时遗失了当时的数据库数据。

这意味着当前仓库可以展示完整代码结构、技术栈、架构设计和实现思路，但无法在没有补充数据库的情况下直接还原毕业设计时期的完整运行效果。

## 学号、账号与推荐身份的关系

代码中可以看到两个相关概念：

- `user.account`：系统登录账号，登录接口使用该字段和密码完成认证。
- `user.certId`：学生身份标识，对应学校学生信息表中的学号。

在原始毕业设计数据场景中，真实学号是学生身份的唯一标识。注册逻辑会校验 `reader.CERT_ID` 是否存在，并防止同一个学号被重复绑定。后端在记录用户浏览、收藏、借阅等行为时，也会把 `certId` 作为用户行为 key 写入 Kafka，后续 Spark 推荐计算同样围绕 `certId` 展开。

需要注意的是，代码层面并不是所有用户都强制要求 `account == certId`。对于非本校学生或普通用户，注册逻辑会将 `certId` 回退为 `account`。因此更准确的表述是：本校学生通过真实学号进行身份绑定，推荐链路以 `certId` 作为用户唯一标识；登录认证层使用 `account` 字段。

## 图书封面补全背景

原始图书馆馆藏数据中包含真实图书信息，但部分图书存在封面缺失或其他信息不完整的情况。`crawler` 子项目即用于根据图书标题从外部网站搜索并补全 `book.img` 字段，使前端图书列表和详情页能够展示封面。

从 `crawler/main.py` 可以看到，爬虫读取 `book` 表中 `img is null` 的图书标题，使用 Selenium 打开当当网搜索结果页，提取图书图片 URL 并回写到 MySQL 的 `book.img` 字段。

## 作为简历项目的定位

这个项目适合作为简历和面试材料展示，重点不应只放在页面效果，而应突出以下能力：

- 能把传统业务系统和大数据推荐链路组合成一个完整工程。
- 能设计前后端分离的图书业务系统，并接入权限认证和管理功能。
- 能通过 Kafka 采集用户行为，再用 Spark/Hive 进行推荐计算。
- 能基于 Spark Streaming、Hive 分区表和 MySQL 回写结果构建完整的数据处理闭环。
- 能使用 Spring Security、JWT、BCrypt、Knife4j / Swagger 等后端工程能力支撑业务系统。
- 能在前端结合 Element UI 和 ECharts 完成业务页面与统计图表展示。
- 能识别真实业务数据中的缺失问题，并编写爬虫工具做数据补全。
- 能说明历史项目的数据缺失、隐私限制和可复现边界。

后续 README 中可以把该项目明确描述为“本科毕业设计项目 / 图书推荐系统 / 前后端分离 + 大数据推荐链路实践”，这样更利于面试官快速理解项目价值。
