# 数据库说明

当前仓库不包含原始数据库数据，也没有完整 SQL 初始化脚本。该缺失不是代码整理造成的，而是历史项目数据已经遗失，并且原始数据中包含真实学生学号、学籍信息和图书馆馆藏信息，不适合直接公开。

## 缺失原因

本项目是本科毕业设计项目，距离完成时间已经约三年。后续因为工作需要继续使用同一台电脑，在清理电脑数据时遗失了当时的 MySQL 数据库数据。

原始 Windows MySQL 中包含：

- 学校真实学生学号、姓名、学院班级等学籍相关信息。
- 图书馆真实图书馆藏数据。
- 用户借阅、收藏、浏览等行为数据。
- Spark 推荐计算后写回的推荐结果数据。

这些数据即使仍然存在，也需要脱敏后才能公开。

## 对当前仓库的影响

当前仓库可以用于展示：

- 前后端分离系统的代码结构。
- Spring Boot 业务接口和权限认证实现。
- Kafka 用户行为采集方式。
- Spark/Hive 推荐计算流程。
- Selenium 爬虫补全图书封面的方式。

当前仓库暂时不能直接做到：

- 一键还原完整毕业设计运行效果。
- 直接启动后看到完整真实图书和用户数据。
- 直接运行推荐计算并得到原始项目中的推荐结果。

后续如果需要增强可复现性，建议补充脱敏后的建表 SQL、少量演示数据和初始化说明。

## 核心业务库

业务库名称在代码中主要体现为 `library`。后端和爬虫都会连接该库。

### 学生信息表

从后端实体可见，`reader` 表用于保存学校学生信息：

| 字段 | 含义 |
| --- | --- |
| `CERT_ID` | 学号，学生身份主键 |
| `NAME` | 姓名 |
| `DEPT` | 学院班级 |
| `REDR_REG_D` | 登记日期 |
| `REDR_TYPE` | 读者类型 |

注册逻辑会校验学号是否存在于学生信息表，并防止同一个学号重复绑定系统账号。

### 用户表

`user` 表用于系统登录和用户资料：

| 字段 | 含义 |
| --- | --- |
| `account` | 登录账号 |
| `password` | 登录密码 |
| `username` | 用户名 |
| `certId` | 用户绑定的学号或身份标识 |
| `status` | 用户状态 |
| `head` | 头像 |

代码中登录认证使用 `account`，而用户行为和推荐计算主要使用 `certId`。

### 图书表

`book` 表用于保存馆藏图书信息：

| 字段 | 含义 |
| --- | --- |
| `M_CALL_NO` | 图书编号 |
| `M_TITLE` | 书名 |
| `M_AUTHOR` | 作者 |
| `M_PUBLISHER` | 出版社 |
| `M_PUB_YEAR` | 出版年份 |
| `STATUS` | 图书状态 |
| `img` | 图书封面 URL |

爬虫子项目会读取 `img is null` 的图书，并尝试补全 `img` 字段。

### 推荐结果相关表

从大数据模块写库逻辑看，推荐结果会回写到 MySQL，主要包括：

| 表 | 含义 |
| --- | --- |
| `recommend` | 面向用户的个性化推荐结果 |
| `relatedbook` | 与图书相关的相似图书推荐 |
| `newbook` | 新书推荐结果 |

具体表结构需要结合后续补充的 SQL 或根据 Spark 写库字段重建。

## Hive 与中间计算表

`bigdata` 模块会把 Kafka 用户行为写入 Hive，并生成多个中间表：

| 表 | 含义 |
| --- | --- |
| `userlog` | 用户行为原始日志 |
| `step1` | 用户-图书行为权重汇总 |
| `step2` | 用户行为向量中间结果 |
| `step3` | 图书共现或关联计算中间结果 |
| `step4` | 关联分数中间结果 |
| `step5` | 皮尔逊相关系数或相似度结果 |
| `step6` | 用户候选推荐中间结果 |
| `step7` | 推荐排序结果 |

这些表依赖 Hadoop/Hive 环境，不属于单纯的本地 MySQL 业务库。
